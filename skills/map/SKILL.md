# /map — 構造把握

`/branch`（どこに置くか）に続く第 2 ステップ。
**手を動かす前に、現在の構造・既存実装・関連ファイル・危険な文脈を把握する。**
`/translate` より先に走り、構造的な地図を後続ステップに渡す。

## なぜ要るか

AI の最大の失敗モードは、古いファイルの誤読だけではない。**構造を把握せず実装に飛び込む**こと:

- 既存の類似実装を見落として二重実装する
- 入口（UI / API / CLI）を特定せず、間違った場所に手を入れる
- 依存関係を把握せず、意図しない波及を起こす
- プロジェクトの既存パターンを無視して異質なコードを足す
- 古いコード・死んだ分岐を「現行の仕様」と重みづけして誤読する

map は「古いファイルを見るな」ではなく、**「実装前に現状の地図を作れ」**。

> **原則: リポジトリは「現在の正しい状態」だけを表す。歴史は git / PR / issue / ADR が持つ。**

## いつ走るか（軽量ゲート・早期終了あり）

- **コード/ファイルを変更するターン**: Step 1 → 2 → 3。
- **読むだけ/会話のみのターン**: Step 1（現在地確認）だけ。残骸を見つけたら surface のみ。
- **早期終了**: 作業箇所が明白・小規模・残骸シグナルなし → `/map: clean（理由）` の 1 行で完了。**仕事を捏造しない**。

---

## Step 1 — 現在地（常に・安価）

```bash
git log --oneline -10        # 直近に何が起きたか / 前回セッションの止まった所
git status --short --branch   # 未コミット残骸 + remote 未反映（## …[ahead N] / upstream無）= push-candidate 検出
git branch --show-current     # 今どのブランチで何の途中か
```

- プロジェクトに「現在地」ポインタ（`STATE.md` / `PROGRESS.md` 等）があれば、現在フェーズを 1 行で確認する。
- 出力: 現在地サマリを 1-3 行。
- git 管理外なら git 系は省略し、ディレクトリの手がかりで現在地を 1 行。

### 作業単位チェック

Step 1 では、未コミット変更や branch 状態が「このまま進める作業中」か「一度 commit / push / merge に出す候補」かを軽く見る。
これは **detection（検知）のみで、execution（実行）ではない**。`/map` 自身は commit / push / PR / merge を行わない。作業が大きくなりすぎたり前回タスクの変更が混ざったりする前に surface して、agent が手を入れる前に一度立ち止まるためのものである。必要なら GitHub workflow に進める候補として短く surface するだけで、自動実行はしない。

確認すること:

- `git status --short --branch` に未コミット変更があるか、`## …[ahead N]` で local commit が remote 未反映か（fetch しないので behind は最後の取得時点）
- 未コミット変更が今回のユーザー依頼と同じ作業か
- 複数テーマの変更が混在していないか
- すでに意味のある一区切りになっていないか
- branch 上の作業が remote へ push / PR / merge に進める候補か

判定:

- `continue` — 今回の作業中として自然。このまま `/map` を続けてよい。
- `commit-candidate` — 変更が一区切りに見える、または今回依頼と別テーマの変更が残っている。先に commit するかユーザーに確認する。
- `push-candidate` — local commit が remote に未反映に見える。ユーザー承認または明示依頼があれば push を検討する。
- `merge-candidate` — branch 上の作業が完了に見え、未コミット変更がなく、PR / merge に進める候補。ただし M2 は CI / review の代替ではないため、最終判断はユーザー・CI・レビューに委ねる。

出力例:

```text
作業単位: continue — 未コミット変更なし。今回の作業に進める。
作業単位: commit-candidate — README と hook の変更が一区切り。次の変更前に commit 推奨。
作業単位: push-candidate — local commit が remote 未反映。明示依頼があれば push 検討。
作業単位: merge-candidate — working tree clean。branch の目的は完了候補。CI / review 後に merge 検討。
```

## Step 2 — 地図作成（変更を伴うターンのみ）

ユーザーの依頼内容から作業箇所を特定し、以下 **6 観点** で地図を作る。
全リポジトリを毎回舐めない — 作業に関係する範囲だけ。

| # | 観点 | 確認すること | 手段の例 |
|---|------|-------------|----------|
| 1 | **現在地** | この機能/モジュールは今どこに実装されているか | grep / find / ファイル構造 |
| 2 | **入口** | UI / API / CLI / config / hook のどこからこの処理に到達するか | 呼び出し元の追跡 |
| 3 | **依存** | 変更がどのファイル・関数・設定に波及するか | import/require の追跡、型定義 |
| 4 | **既存パターン** | 似た実装はどこにあるか。プロジェクトの慣習は何か | 類似コードの確認 |
| 5 | **触らない場所** | 今回の依頼と関係ない領域はどこか（blast radius の境界） | ユーザーの依頼スコープ外を明示 |
| 6 | **危険な文脈** | old/backup/archive/tmp/legacy を現行と誤読していないか。残骸シグナルはあるか。前回プロンプトがあれば: その作業痕跡が working tree / git log に存在するか | 後述の残骸シグナル表で走査 + 前回タスク痕跡の確認 |

### 出力形式

```
## /map Step 2: 地図

1. 現在地: `src/handlers/call.ts` — 通話処理のメインハンドラ
2. 入口: Twilio webhook → POST /voice → handleIncomingCall()
3. 依存: src/utils/tts.ts, src/config/prompts.ts, .env(API_KEY)
4. 既存パターン: 他のハンドラ（sms.ts, status.ts）は同じ Express router パターン
5. 触らない場所: src/billing/, src/auth/ — 今回のスコープ外
6. 危険な文脈: なし（残骸シグナル検出ゼロ）
```

小さな変更なら 6 行で済む。大規模変更なら各項目に複数行を使う。
**6 観点を埋められない項目がある場合は「不明（要調査）」と書く。空欄で飛ばさない。**

## Step 3 — 残骸の始末（Step 2 観点 6 で検出された場合のみ）

Step 2 の観点 6 で残骸シグナルが **ゼロなら Step 3 はスキップ**。

### 残骸シグナル

| シグナル | 例 |
|----------|-----|
| コメントアウトされた実装 | `// function oldX() {...}` のブロック |
| 死んだ分岐・併存実装 | `oldImpl()` `newImpl()` `newImplV2()` の並存 |
| 残骸ファイル | `*_old` `*.bak` `tmp*` `backup*` `*.orig` |
| 食い違う文書 | 現コードと矛盾する README / コメント / 仕様メモ |
| 古い TODO/FIXME | 既に解決済み・前提が消えた TODO |
| 重複モジュール | ほぼ同じ処理の二重実装 |
| 亡霊参照 | 削除済みファイル / 実在しない SSOT を指す記述 |

### 3 問ゲート（DELETE / SURFACE の判定）

残骸を消そうとするたび、**1 件ごと** にこの 3 問を通す。
**1 つでも NO / 不明 → SURFACE**（触らずユーザーに提示）。**全部 YES → DELETE**。

1. **自作か？** — 自分（または現行コード/ビルド）が生成したものか？
   - NO / 不明 → SURFACE
2. **保護カテゴリ外か？** — 下記のどれにも当たらないか？
   - 1 つでも当たる → SURFACE
3. **参照ゼロか？** — grep で参照ゼロを確認できたか？
   - 確認できない / "古く見える" だけ → SURFACE。"古さ" は証拠ではない。

### 保護カテゴリ（触らず SURFACE）

- 自作でない / 由来不明
- 外部ツール由来（別の AI/CLI が置いた設定・指示ファイル）
- CI / build / lint / パッケージ設定
- vendor / 生成物 / ロックファイル
- 鍵 / 環境変数 / テンプレート（`.env*` / `*.key` / `*.pem` / `credentials*`）
- 宣言された SSOT / live 参照 / 凍結アーカイブ

### 始末の原則

1. **3 問全 YES → 証拠ベースで削除**（承認不要）。確信できる残骸を過度に残さない。
2. **どこかで詰まった → SURFACE**（推奨を添えて提示。迷ったら消さない）。
3. **今も要る文脈は「現在形の理由」に変換** して 1 行で残す。死体（旧コード）は残さない。
4. **旧実装を残したまま新実装を隣に足さない**（増殖アンチパターン）。

---

## 他の門との関係

- これは **偵察と整地であって破壊許可証ではない**。大量削除に化けさせない。
- 過去セッションが消した「改善」を、整地の名目で復活させない。
- 削除を伴うなら blast radius は `/branch` が引き続き守る。

## 禁止事項

- 地図を作らずに実装に飛び込む（Step 2 を省略して「だいたいわかった」で進めない）
- 3 問ゲートを通さない自動削除
- 証拠なしの削除（"古く見える" だけで消す）
- 旧実装を残したまま新実装を隣に足す（増殖）
- 残骸を「念のため」で残す（証拠があるなら過度な保守も失敗。履歴は git にある）
- 仕事の捏造（clean なら 1 行で終える）
- ユーザー承認や明示依頼なしに commit / push / merge しない
- CI / test / review を見ずに merge 可能と断定しない
- 複数テーマの変更を 1 commit にまとめる判断を勝手にしない

## 完了時の処理

終了報告の前に、`.agent-harness/current-task.json` の `phase.map` を `done` に更新し、`.agent-harness/current-task.md` に 20 行以内の地図 summary を保存する:

```json
{
  "schema": 1,
  "task_id": "20260618-abc123",
  "branch": "feat-login",
  "prompt_hash": "sha256:...",
  "phase": {
    "branch": "done",
    "map": "done",
    "translate": "pending"
  },
  "scope_hash": "",
  "updated_at": "2026-06-18T12:34:56Z"
}
```

`map-guard` hook は空ファイルではなく、この state が現在の branch / prompt と一致し、`phase.map = done` であることを検証する。
`/branch` が新しい task state を作り、旧版の空 sentinel が残っていれば削除する。

## 終了報告フォーマット

```
/map: <clean | 地図サマリ + DELETE n 件 / SURFACE m 件>
```
