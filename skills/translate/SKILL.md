# /translate — 段階翻訳スキル（triage 付き・Step 0 常時／Step 1-3 は必要時のみ）

ユーザーの指示を **コードに触る前に** 翻訳する。まず下の Triage で仕分け、必要な段だけ実行する。
「理解したつもり」での破壊を防ぐのが目的。

```
Step 0: 用語定義確認 ── 曖昧な動詞・名詞・修飾語を抽出し、ユーザーに定義を聞く
  ↓ （ユーザーの回答を待つ。勝手に Step 1 へ進まない）
Step 1: 論点抽出 ────── 発言から全ての指示を漏れなく列挙
Step 2: エンジニア語 ── 各論点を技術的な問題文に変換
Step 3: 具体箇所 ────── 各論点をリポジトリのファイル・関数・行に紐付け
```

## Triage（最初に1回・数秒）— 全部を毎回やらない

重い門は飛ばされ、肝心の Step 0 が死ぬ。だから **まず仕分けて、必要な段だけ実行する**:

- **A. 読むだけ／会話／情報提供／既翻訳と同一論点の継続** → **Step 0 の曖昧語スキャンだけ**。曖昧ゼロなら「Step 0: 検出ゼロ（理由）」の1行で完了。Step 1-3 は不要。
- **B. コード／ファイルを実際に変更する** → **Step 0 → 1 → 2 → 3**。

要点: **Step 0 は常に走る軽量ゲート**。Step 1-3 は B のときだけ。

---

**Step 0 は最優先・必須**。曖昧語を 1 つでも検出したら、Step 1 以降に進む前にユーザーに確認する。

---

## Step 0: 用語定義確認【必須・最優先】

### 目的

同じ単語でも解釈が複数あると、AI は勝手に最もそれらしいものを採用して手を動かす。
**「テスト」「チェック」レベルの日常動詞を、作業内容・基準・結果の扱いまで単語単位で詰める** のが Step 0。

### 検出対象

以下のいずれかに該当する語・表現を発言から抽出する。

#### A. 曖昧な動作動詞（最重要）

| 動詞 | 確認すべき観点（3 点セット） |
|------|-----------------------------|
| テスト / 試す / 動作確認 | ① 何を実行するか（ユニット `npm test` / 型チェック `tsc` / lint / E2E / 手動ブラウザ確認 / CIで実行されるもの全部） ② 対象スコープ（変更ファイルのみ / 関連ディレクトリ / 全体） ③ 合格基準（全 green / エラー 0 / 既存と同じ状態） |
| チェック / 確認 / 見て | ① **何を** 確認するか（動作 / 型 / 実装の正しさ / 副作用の有無 / 他箇所への影響） ② **どの基準で** 判定するか（仕様書 / 過去の挙動 / 似た処理との一貫性） ③ **結果をどうするか**（報告のみ / 自動修正 / 中断して相談） |
| 修正 / 直す / 対応 / 解決 | ① 症状除去か根本修正か ② 関連箇所にも波及させるか（1 箇所だけ / 同種バグ全部 / 再発防止策まで） ③ 互換性保持の要否 |
| 調査 / 調べる / 見ておいて | ① 読むだけか変更も許容か ② 報告形式（1 行要約 / 詳細 / 提案付き） ③ 時間・範囲の上限 |
| 動かす / 実行 / 回す | ① 1 回動けば OK か継続運用か ② ローカルのみか環境込みか ③ 失敗時のロールバック方針 |
| 整理 / リファクタ / 綺麗に / 片付ける | ① **何を基準に** 綺麗とするか（コード規約 / 短くする / ファイル分割 / 命名統一） ② 削除可否・非互換変更の可否 ③ テスト境界 |
| レビュー | ① コメントのみか変更も入れるか ② 観点（セキュリティ / 性能 / 可読性 / 全部） |
| 追加 / 実装 / 作って | ① 既存拡張か新規機能か ② テスト・ドキュメント・型の付随有無 ③ UI / API の仕様 |
| 消す / 削除 | ① 論理削除か物理削除か ② 呼び出し元の扱い ③ テスト・ドキュメント・履歴も消すか |

**3 点セット**（①何を／②基準は／③結果をどうするか）を毎回埋めさせる。
「テスト」と一言で終わらせない。

#### B. 曖昧な対象（指示語・抽象語）

- 指示語（それ・これ・あれ・この・その・あの）で、直前の文脈に **明示されていない** 参照先
- 「末尾」「先頭」「該当箇所」「あの部分」「例のやつ」等、**境界が不明** な位置・範囲表現
- 「ユーザー」「顧客」「アカウント」「マスター」等、**ドメイン内で定義が分かれる** 名詞
  （顧客 = 企業 or 個人？アカウント = ログインID or 課金単位？）

#### C. 程度・品質の曖昧な修飾語

- ちゃんと / きちんと / しっかり / 適切に / いい感じに / うまく / 綺麗に / それっぽく / 普通に
  → **合格条件を必ず聞く**。「ちゃんと動く」の「ちゃんと」の定義を詰めない限り検収は成立しない。

#### D. スコープ不明な語

- 「全部」「全体」「関連する箇所」「必要なところ」「他のも」「ついでに」
  → **具体的な対象リスト**（ファイル名・関数名・機能名）に落とし込む。
- 「関係ありそうなところ」「まとめて」「全部」「必要なら」は scope expansion phrase として扱う。
- 「急いで」「本番で困っている」は優先度シグナルであり、gate 省略や scope 拡大の許可ではない。

#### E. 主語（操作対象）の省略【省略不可】

動詞が明確でも、**何に対して操作するかが文中に存在しない**場合。
操作対象とは: ファイルパス・コンポーネント名・プロジェクト名・リポジトリ名のいずれか。

- 典型例: 「直して」「追加して」「消して」「反映して」だけで対象が無い
- 暗黙の継続例: 「devのも」「あのプロジェクトも」→ B（指示語）と重複することがあるが E でも必ず確認
- 文脈から推測できても**推測で進めない**。候補を提示して選ばせる

確認形式（他の Step 0 項目と同じフォーマットで出力）:
```
### 「<動詞>」の操作対象
- 対象（何に対して）:
  - (a) <文脈から最有力の具体パス・名前>
  - (b) <別候補があれば>
  - (c) その他（具体的に教えてください）
```

### 検出と確認のルール

1. **検出ゼロの宣言も必須**。曖昧語なしと判断した場合も「Step 0: 検出ゼロ（根拠: すべて具体的なファイル名と動作が指定されている）」と 1 行で明示する。無言でスキップしない。
1a. **E（主語省略）は A-D と独立して必ず確認する**。A-D の曖昧語がゼロでも、操作対象（ファイル・コンポーネント・プロジェクト）が文中に具体的に明示されていなければ E として確認する。
2. **1 つでも検出したら Step 1 以降に進まない**。ユーザー回答を待つ。
3. **複数の候補を提示する**。最も可能性が高いものを (a) に置く。ユーザーが「a で」と短縮回答できるように。
4. **「たぶんこれだろう」で勝手に埋めない**。推測で進めるくらいなら聞く。
5. **ユーザーが「全部お任せ」「そのまま進めて」と返したら**、各語について AI エージェントが採用する解釈を一覧で提示してから Step 1 に進む（事後承認型）。

### Step 0 出力テンプレート

```
## Step 0: 用語定義の確認

以下の表現について、進める前に定義を教えてください。
（回答が揃うまで Step 1 以降には進みません）

### ① 「<原文の曖昧語>」
- **① 何を** :
  - (a) <候補1。最も可能性が高いもの>
  - (b) <候補2>
  - (c) <候補3>
- **② 基準**: <判定基準を聞く質問。例: 成功とみなす条件は？>
- **③ 結果**: <結果の扱いを聞く質問。例: 失敗時は自動修正？報告のみ？>

### ② 「<別の曖昧語>」
...（同様に列挙）

---
短縮回答可: 「全部(a)でOK」「お任せで」「① だけ (b) で、残りお任せ」等
```

### Step 0 実例

ユーザー発言: 「通話テストしてチェックして、あとあれも直して」

```
## Step 0: 用語定義の確認

3 つの曖昧な表現を検出しました。

### ①「通話テスト」
- 何を:
  - (a) `npm run test:voice` を実行（ユニットテスト）
  - (b) ローカルで Twilio webhook を叩いて実通話シミュレーション
  - (c) 本番番号に実電話
- 基準: どこまで green なら完了？（全テスト通過 / 特定シナリオだけ / エラー 0）
- 結果: 失敗時は自動修正を試みる？それとも報告のみで止まる？

### ②「チェック」
- 何を: テスト結果を確認するのか、コード実装の正しさを確認するのか、両方か
- 基準: 何と突き合わせて判定する？（仕様書 / 既存挙動 / 私の過去指示）
- 結果: 問題が見つかったら修正まで進む？それとも報告だけ？

### ③「あれ」
- 「あれ」の参照先が不明です。直前の会話に該当しそうなのは:
  - (a) 昨日の「エラーメッセージの文言」
  - (b) 先ほどの「ログイン画面」
  - (c) それ以外（具体的に教えてください）

---
短縮回答可: 例「① a、② 両方確認して報告のみ、③ b」
```

### glossary による省略（オプション）

プロジェクト固有の用語定義があれば `.agent-harness/glossary.md` に書いておく。
該当語は Step 0 でユーザーに確認せず glossary の定義を採用できる（ただし「glossary の X を採用しました」と 1 行宣言する）。

例: `.agent-harness/glossary.md`
```
- テスト: `npm test` を実行。red なら修正を試行し再実行（最大 3 回）。3 回失敗したら中断してユーザーに報告。
- チェック: 型エラー（`tsc`）と lint（`eslint`）を実行。エラーがあれば自動修正してから報告。
- レビュー: コードを読み、問題点をコメントで列挙するのみ。変更は加えない。
```

## Step 1: 論点抽出

Step 0 で用語が確定した後、発言から全ての指示を漏れなく列挙する。

### ルール

- **感情表現（怒り・皮肉・強調・あきらめ）は論点の優先度と重大度のシグナル**。無視しない。
  - 例: 「またかよ…」→ 再発案件。再発防止策の検討が追加論点になる。
- **「ついでに」「あと」「それと」「ちなみに」の後続も独立した論点** として扱う。
- **省略・統合・要約しない**。ユーザーが 3 つ言ったら 3 つ出す。似ているからまとめて 1 つにしない。
- **疑問文**（「○○ってどうなってる？」）も情報提供の論点として拾う。
- **対比表現**（「A じゃなくて B」）は A（禁止事項）と B（要求）の 2 論点に分解。
- **否定形**（「○○しないで」）は明示的な制約として記録する。

### 出力形式

```
## Step 1: 論点抽出

- 論点 1: 「<原文引用>」 → <要約>
- 論点 2: 「<原文引用>」 → <要約>
- 論点 3: 「<原文引用>」 → <要約>
- 制約 1: 「<否定形の原文>」 → <禁止事項として記録>
```

## Step 2: エンジニア語への変換

各論点を技術的な問題文と仮説に翻訳する。

### 変換の型

| ユーザー語 | エンジニア語の型 |
|------------|------------------|
| 「○○がおかしい」 | 「○○の処理（関数/モジュール）にバグがある可能性。入力・出力・境界条件を検証」 |
| 「表示が変」 | 「UI レンダリング or 状態管理の問題。コンポーネント・props・state を確認」 |
| 「動かない」 | 「エラー発生 or 期待動作なし。エラーログ・スタックトレース・再現手順を確認」 |
| 「遅い」 | 「パフォーマンス問題。該当処理のプロファイリング。N+1・同期 I/O・大量データを疑う」 |
| 「○○を追加」 | 「新機能実装。入出力仕様と既存コードとの統合点を定義」 |
| 「戻らない」 | 「状態遷移バグ。terminal state 判定 or イベント発火条件を検証」 |
| 「表記がおかしい」 | 「文字列生成ロジックの不整合。テンプレート・変数展開・フォーマッタを確認」 |

### 出力形式

```
## Step 2: エンジニア語

- 論点 1: <エンジニア語の問題文> / 仮説: <技術的仮説>
- 論点 2: <エンジニア語の問題文> / 仮説: <技術的仮説>
```

---

## Step 3: リポジトリ具体箇所

各論点を具体的なファイル・関数・行に紐付ける。
`/map` Step 2 が既に構造を把握しているため、その地図を活用する。

### 手順

1. `/map` Step 2 の地図（現在地・入口・依存・既存パターン）を参照し、論点との対応を確認
2. 地図でカバーされていない箇所は `Grep` / `Read` で補完
3. 該当行を特定し `path:line` 形式で記録
4. **1 本に絞れないときは複数候補を提示する**（絞ったせいで外すのが最悪）

### 出力形式

```
## Step 3: リポジトリ具体箇所

- 論点 1: `src/server.ts:45 buildInstructions()` の日付フォーマット処理
- 論点 2: `src/handlers/recordMessage.ts:49` の TTS 文字列生成
- 論点 3: 候補A `src/utils/parse.ts:12` / 候補B `src/core/normalize.ts:88`（どちらか特定要）
```

---

## 完了時の処理

終了報告の前に、`.agent-harness/current-task.json` の `phase.translate` を `done` に更新し、`scope_hash` も今回の scope から更新する。
**Step 3 で特定したファイルを `scope.files` 配列として記録する**（4 つ目のゲート `scope-guard` がこれを使って範囲外編集を block するため）:

```json
{
  "schema": 1,
  "task_id": "...",
  "branch": "...",
  "prompt_hash": "...",
  "phase": {
    "branch": "done",
    "map": "done",
    "translate": "done"
  },
  "scope": {
    "files": [
      "src/routes/login.tsx",
      "src/components/LoginForm.tsx"
    ],
    "new_files_allowed": false
  },
  "scope_hash": "sha256:...",
  "updated_at": "..."
}
```

- `scope.files`: Step 3 で確定したファイルパス（リポジトリルートからの相対）の配列。glob (`src/auth/*.ts`) も可。
- `scope.new_files_allowed`: スコープに無い**新規**ファイル（ディスク上に存在しない）の作成を許すか。デフォルト `false`。新規ファイル作成を伴うタスクのみ `true`。

あわせて `.agent-harness/current-task.md` を 20 行以内に保ち、scope / out of scope / next を短く保存する:

```md
# Agent Harness M2 Current Task

Branch: feat-login
Task: ログイン画面を追加する

Map: done
Translate: done

Scope:
- src/routes/login.tsx
- src/components/LoginForm.tsx

Out of scope:
- billing
- admin

Next:
- Implement within scope
```

`translate-guard` hook は `current-task.json` が現在の branch / prompt と一致し、`.agent-harness/.gates/branch.done` と `.gates/map.done` の platform-signed sentinel が存在し、`phase.translate = done` であることを検証する。
`scope-guard` hook は `phase.translate = done` で `.gates/translate.done` が存在することを確認したうえで、`scope.files` 外のファイルへの Edit/Write を block する。**スコープ外を触りたくなったら必ず `/translate` を再実行してスコープを更新する。** 黙って広げない。

## 完了宣言

実行した Step の出力を揃えたうえで、実装に入る前に必ず以下を宣言する:

```
/translate 完了。以上で理解が合っていれば実装に進みます。
ズレがあれば指摘してください。
```

宣言後、ユーザーの OK を待ってから実装に入る。
ただしユーザーが事前に「そのまま進めて」と明示している場合はこの限りでない。

## 絶対ルール（繰り返し）

- **Step 0 の用語定義確認をスキップしない**。曖昧語を「たぶんこれだろう」で解釈して進めない。
- **操作対象（主語）が省略されていたら必ず確認する**（E）。文脈から推測できても推測で進めない。
- **Step 1 以降でユーザーが 3 つ言ったら 3 つ出す**。統合・要約しない。
- **Step 3 で 1 本に絞れないときは複数候補を出す**。推測で絞らない。
- **完了宣言の後、実装 OK をもらうまで手を動かさない**（事前承認済みを除く）。
